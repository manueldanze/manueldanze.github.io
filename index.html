    Project-Thomas

      A puzzle platformer prototype, inspired by the game 'Thomas Was Alone' by Bithell Games. Created with Unity 2022.3.11f1.

      Developed as a solo project during my 5th semester of Computer Science Media at Stuttgart Media University in the Winter Semester of 2023/24.

      Difficulties
      • This was my first time developing a game using the Unity Engine, so I had to learn about specific Unity classes and functionalities like MonoBehaviour and Scriptable Objects.
      Improvements
      • I was able to utilize my experience from other projects to find a fitting scale and dimension for the prototype right from the beginning.
      • The entire development went quite smoothly after I became confident with the idea of making a 'clone' of an existing game. Having a reference point helped me work effectively as a solo developer.
      Learnings
      • I learned a lot about Unity-specific functionalities.
      • I learned that, with proper planning, it is possible to deliver a good product within the schedule without crunching or being stressed out
      • Although this solo development experience was smooth, there were often times when I wished for feedback or the ability to share ideas with team members.

      Project Massive

        A third-person, online-multiplayer playground prototype. Created with the Unreal Engine 5.2 (Sourcecode Build) using UE5 Blueprints. The game server was hosted on a Kubernetes Cluster on Google-Cloud.

        Developed as a software project by 3 students over a period of 4 months, in the 4th semester of Computer Science Media at Stuttgart Media University.

        My Contribution
        • The idea of making a UE5 MMO Game based on a flexible containerized server structure
        • Project management e.g organization and documentation via monday.com
        • Curation of the used version control system Perforce Helix Core
        • Built and made sure the UE5 Sourcecode Build Editor Version worked for every team member
        • Grabbing/throwing mechanic
        • Textchat mechanic
        • Score system and the collision/overlap handling
        • Distinction and assigment of points to a certain player on the server
        • Networking of all systems and build/packaged the gameserver
        • Basketball quest, the glowing orb quest and the scoreboards
        Difficulties
        • To build/package a game server and make use of the networking abilities of the engine, it was neccessary to build the UE5 editor from the free available sourcecode. This process was complicated, time consuming and occupied an enormous amount of disk space
        • The team had no experience with big game engines at all, so we figured out how to use the UE5 Editor and Blueprints via learning-by-doing
        • We never used a centralized version control system and only had little experience with Git. So using Perforce and UE5 challenged our team quite a bit
        • Due to lack of time and ressources we had to simplify systems and ideas. One team member was exclusively working on the complicated containerization process which made the workload distribution difficult
        • The whole project was very ambitious for 3 students and only one semester of time. It took a lot of effort to find the right scale
        Improvements
        • Due to lack of time and know-how we had to use several workarounds like the score system in which we hardcoded the first three players on the scoreboard. This should be avoided and replaced by a rotation through all players on the server
        • We also could have scaled back parts of the project, for example the containerization to free up more resources. With more time and/or team members we could have achieved the project's goal more effectively
        Learnings
        • I gained experience with UE5 Blueprints programming and the network functionalities of the editor
        • I had many difficulties with the editor, Perforce as version control system and the whole Engine building process, I learned a lot while fixing these problems
        • It is very important, that at the start of a project the scale of the project itself matches the amount of ressources the team can afford. Sometimes it is necessary to cut parts and scale the project down in order to be able to fulfill milestones and deliver the original vision.

        Game Master

          A topdown bullethell-arena local-multiplayer with a Game Master mechanic. Created using Java, LibGDX and Box2D.

          Developed as a software project by 6 students in 3rd semester of Computer Science Media at Stuttgart Media University.

          My Contribution
          • Game idea of a bullet-hell arena multiplayer
          • Logic of the menu screens
          • Connection of different game systems with the Kryonet networking part
          • Logic of ingame screen overlays
          • Object collision and overlap handling via Box2D physics library
          • Loading sprites from a spritesheet
          • Simple CI/CD pipeline in Gitlabs
          • I suggested using SCRUM as part of an agile software development principle for this project. I took responsibility for the Scrum Master position, moderated the meetings and curated the Gitlab ticket system "Issues", the changelog and readme files
          Difficulties
          • It was a challenge to keep all 6 team members focused and concentrate the essence of all ideas into the original vision of the game without changing the whole concept
          • We had issues with different ideas for the game design of the core gameplay, which in the end did not work out as planned and ended up in a non functional gameplay loop
          • We also ran into big issues trying to implement the Kryonet Networking library and the Box2D physics library, halfway during the development and not from the start
          Improvements
          • A good software architecture and an idea of the necessary libraries and systems right at the beginning, could have been very helpful to avoid the time consuming changes during the development
          • Better communication and idea of the basic gameplay would have lead to a more satisfying development process and final result
          Learnings
          • I learned how to use libraries and frameworks and made my first experiences with LibGDX and Box2D
          • I experienced the importance of interesting and consistent game design in order to keep the gameplay loop working
          • I recognized how crucial good planning of the architecture, concept and used libraries is
          • I got used to work with a ticket system, how to curate a changlog and set up a CI/CD pipeline
